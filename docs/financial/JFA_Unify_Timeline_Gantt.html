📍 Detalhe por Fase

Fase 1: MVP — Tuya Hub + PIN Pad (Semanas 1-2)

Deliverable: Backend FastAPI + SvelteKit PIN UI + First pilot location
  • W1: Hardware sourcing (Tuya distribuidor), FastAPI skeleton, MQTT broker setup
  • W1: Keycap integration (Tuya API), PIN validation (4-12 digits), rate-limiting
  • W2: Unit tests (Jest/Pytest), integration tests (MQTT TLS handshake)
  • W2: Pilot #1 deploy (WC-A), field testing, gather feedback

Fase 2: Card Reader — NFC/RFID PN532 (Semanas 3-4)

Dependency: PN532 board arrival (+3 dias após order W1)
  • W3: PN532 driver integration, card UID parsing (protocol validation)
  • W3: Double-auth flow (PIN → Card → Relay actuate)
  • W4: Integration tests, e2e testing (Selenium PIN pad + card reader mock)
  • W4: Pilot #2 & #3 deployment (2 additional locations)

Fase 3: JFA_Suite Integration + RLS Auditoria (Semana 7)

Critical Dependency: Suite RLS schema merge (coordenado com Suite lead architect)
  • W7: access_control.input_devices table design (schema review)
  • W7: RLS policies (multi-tenant isolation, row-level access)
  • W7: API endpoints + validation, auditoria logging

Fase 4: JFA_AccessPay Payment Integration (Semanas 8-9)

Dependency: Suite API stable (Phase 3 merged)
  • W8: Stripe webhook handler (HMAC signature verification), event routing
  • W8: Local cache + retry logic (failover pre-auth stored-value)
  • W9: Moloni API integration (invoice generation, SAF-T export)
  • W9: Staging testing (100% coverage payment flow)

Fase 5: JFA_Remotes Gates + Production Hardening (Semanas 10-11)

Deliverable: Full stack production-ready (gates, remotes, monitoring)
  • W10: Shelly relay + Tuya relay bridge, E2E testing (all 3 projects)
  • W10: Production hardening (structured logging, error monitoring, alerting)
  • W11: Deploy runbooks (dev/staging/prod checklists), ops docs
  • W11: Rollout procedure (blue-green deploy, failover tests)

⚠️ Riscos Críticos do Caminho

1. Hardware Lead Time (W1)

Risco: Tuya distribuidor delay → 2 semanas atraso timeline
Impacto: Phase 1 start blocker
Mitigation:
  • Pre-order W0 (antes da sessão PLANEAR)
  • Backup distribuidor secundário (Farnell)
  • Negotiate expedited shipping (€50 extra)

2. Suite RLS Schema Complexity (W5-W6)

Risco: Suite architect redesign → merge blocker, +2 semanas
Impacto: Phase 3 start delay, cascading to Phase 4-5
Mitigation:
  • Pair programming with Suite architect (W2 kickoff)
  • Schema POC/review W2-W3 (antes de W7 merge)
  • Contingency: local input_events table (if Suite schema blocked)

3. Pilot #1 Feedback Loop (W2)

Risco: WC-A rejeita PIN-only, demanda card-only or different UX
Impacto: Phase 2 pivot, design change
Mitigation:
  • User testing W1 (UI mockup feedback antes de deploy)
  • Flexible PIN validation (4-8-12 digit options configurable)
  • Quick pivot capability (feature flags)

4. Payment Handler Complexity (W8-W9)

Risco: Stripe webhook + Moloni API integration = testing overhead
Impacto: E2E testing W10 delayed, risk of production bugs
Mitigation:
  • Stripe test mode (W8 integration testing)
  • Moloni sandbox API (W8 testing)
  • 100% unit + integration test coverage requirement
  • Staging validation 3-5 dias antes W10
